# /review — 브랜치 리뷰 (글로벌)

> 프로젝트에 별도 `/review`가 있으면 그쪽을 먼저 써라. 이건 폴백.

## 1. 변경 요약
- `git diff main...HEAD` (또는 현재 베이스)로 전체 diff를 읽어라.
- 변경의 "의도"를 한 문장으로 요약. 커밋 메시지를 신뢰하지 말고 diff로 직접 검증.

## 2. 공통 체크리스트

### 정확성
- 의도한 동작을 실제로 하는가? 엣지 케이스 (null, 빈 배열, 0, 음수, 매우 큰 입력)
- 에러 처리: 조용히 삼키는 catch 없나? 로그/재throw 적절한가?

### 가독성·설계
- 이름이 의도를 드러내는가? (`data`, `result`, `tmp` 같은 무의미 네이밍 지적)
- 한 함수가 한 가지만 하는가? 깊이 3단계 이상 중첩은 빼낼 수 있는지 검토.
- 주석은 "왜"를 설명하는가? "무엇"만 적힌 주석은 불필요.

### 테스트
- 새 동작에 테스트가 있는가? 없으면 어떤 케이스가 빠졌는지 지적.
- 테스트가 구현 세부(내부 함수명, 상수)를 스누핑하지 않는가?

### 보안·개인정보
- 사용자 입력이 SQL/쉘/HTML에 들어가는 경로에서 이스케이프/파라미터화 됐는가?
- 시크릿이 로그, 에러 메시지, URL 파라미터에 새지 않는가?
- 외부 의존성 추가 시 라이선스·유지보수 상태 확인했는가?

### 성능
- N+1, 불필요한 재계산, 과도한 리렌더 / 리페치가 있는가?
- 비동기 작업이 직렬로 돼 있는데 병렬화 가능한 곳이 있는가?

### 의존성·호환성
- 공개 API / 공용 타입이 바뀌었다면 호출부 전부 확인했는가?
- Breaking change면 마이그레이션 경로가 문서화됐는가?

## 3. 출력 형식

각 지적사항을 다음 중 하나로 분류해서 묶어라:

- 🔴 Must — 머지 전 반드시 수정
- 🟡 Should — 지금 안 바꿔도 되지만 근시일 내 처리
- 🟢 Nit — 취향·일관성 이슈

각 항목에는 **파일:라인**과 **제안**을 포함.

## 4. 마무리
"머지 승인 가능" / "Must 해결 후 재리뷰" 중 하나로 명확히 결론.
