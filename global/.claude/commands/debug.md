# /debug — 가설-검증 디버깅

**규칙**: 수정 코드를 쓰기 전에 **원인을 특정**하라. 추측 패치 금지.

## 1. 증상 정리
- 재현 조건 (입력, 환경, 순서)
- 기대 동작 vs 실제 동작
- 에러 메시지 / 스택 트레이스 전문 (접지 말 것)
- 언제부터 발생? 마지막으로 잘 되던 커밋이 있나?

## 2. 증거 수집 (코드 수정 전)
- 관련 함수·파일을 `Read`로 확인.
- `grep`/`rg`로 관련 심볼·에러 문자열 추적.
- 의심 지점이 "최근 커밋에 손 댄 곳"인지 `git log -p -- <file>`로 확인.
- 필요하면 `git bisect` 전략 제안 (자동화하지 말고 사람이 돌리도록).

## 3. 가설 세우기
**최소 2개**의 가설을 세우고 각각에 대해:
- 이 가설이 맞다면 어떤 증거가 더 나와야 하나?
- 이 가설을 **반증**할 수 있는 실험은 뭔가? (로그 추가, 특정 입력 테스트)

가설을 늘어놓기만 하지 말고 **확률이 높은 순**으로 정렬.

## 4. 검증
- 가장 확률 높은 가설부터 실험.
- 실험 결과가 가설을 지지하는지 반증하는지 명확히 기록.
- 반증되면 가설 목록에서 지우고 다음으로.

## 5. 수정 전 승인
원인이 특정되면 **수정 범위와 접근법**을 먼저 설명하고 승인 요청.
- 근본 원인 수정인가? 증상만 가리는 패치인가?
- 회귀 방지 테스트를 추가할 수 있는가?

## 6. 수정 후
- 원래 증상이 사라졌는지 확인.
- 인접 경로에 부수 효과 없는지 최소 1개 관련 테스트 추가 후 실행.
- `progress.md`에 "원인 — 수정 — 재발 방지" 한 단락으로 기록.
