# /plan — 구현 전 계획 수립

다음 순서로 진행해라. **코드는 아직 쓰지 마.**

## 1. 목표 재확인
- 사용자가 원하는 최종 결과물을 한 문장으로 요약.
- 모호한 부분이 있으면 질문 목록을 만들고 승인 전에 해소.

## 2. 영향 범위 탐색
- 관련 파일을 최소 3개 이상 `Read`로 읽어라. 테스트 파일이 있으면 포함.
- 필요 시 `Explore` 서브에이전트에 위임해서 넓은 탐색을 병렬로 돌려라.
- 현재 의존성·진입점·부수 효과 지점을 파악해서 "수정될 파일 목록" 초안을 만들어라.

## 3. 옵션 제시
- 구현 접근법을 **최소 2개** 제시하고 각각의 장단점, 리스크, 되돌리기 비용을 명시.
- "추천: X (이유)" 한 줄로 끝맺음.

## 4. 실행 계획
다음 형식으로 제시:

- 수정할 파일: `path/to/file.ts` — 무엇을/왜
- 새로 만들 파일: `path/to/new.ts` — 책임은 무엇
- 테스트 계획: 어떤 케이스를 커버하는지, 기존 테스트에 영향 있는지
- 검증 명령어: typecheck / lint / test 중 어떤 걸 돌릴지
- 커밋 전략: 하나로 할지 나눌지, 순서

## 5. 승인 요청
"이대로 진행할까요?" 한 줄로 끝. 승인 전까지 파일 수정·생성 금지.
